# v0.6.0 — Furniture Collision Engine

**發布日期:2026-07-22**

把「家具碰撞」從 *validate 抓到就整份設計失敗*,改成 *有系統地偵測 → 主動修復*,
`validate_spec` 退為安全網。

抽象核心是 **Obstacle**(不是 Furniture vs Furniture):牆、柱、門迴轉、樓梯、
天井、家具都能包成障礙。**新增一種障礙 = 加一個 provider**,detector 與 resolver
不必改——這個設計在 Phase 2~4 連續四次擴充中都成立,沒有回頭重構過。

---

## 主要功能

### 🪑 Furniture Collision(Phase 1)
家具×家具、家具×門迴轉的偵測與修復。偵測範圍刻意與 `validate_spec` 現有檢核
**逐字一致**,確保接進生成流程後對現有合格案例**零改動**(no-op 保證)。
修復原語:`try_move`(沿牆滑動 / 垂直移回房內)、`try_drop`(只丟裝飾家具,
床/馬桶/沙發等必要家具永不丟)。

### 🧱 Wall Collision(Phase 2)
以 **Room Polygon(牆中心線)** 為 barrier,判準是
「碰撞 footprint 突出所屬房間面積 > `WALL_TOLERANCE_MM`」。

關鍵在於**不能用面積重疊判牆**:家具貼牆時 footprint 必然壓到牆的內半邊,
那是合法的。實測佐證:合法貼牆突出 ≤1mm²,真穿牆 ≥170,000mm²,相差五個
數量級,故取 5000mm² 留裕度。桌椅組(table4)另用收緊的
`fixture_collision_footprint`,把「拉開椅子」的範圍與牆演算法分離。

### 🕳️ Void Collision(Phase 3-1)
天井/中庭/挑空為**硬障礙**(`area > OVERLAP_TOL`)——那裡沒有樓板,
壓到多少就錯多少,沒有「貼著算合法」的餘地。

同時補上一個真破口:`_room_polys` 排除 patio,導致**形心落在天井的家具
`room=None`,連穿牆都驗不到**。納入 VOID 後這種家具才會被抓出來。

### 🪜 Stair Collision(Phase 3-2)
梯段為硬障礙,100 層中 **88 座樓梯**納入保護。

接線前先做誤判量測:停用 stair provider 生成全部樓層,再用真實樓梯幾何回頭量,
確認**會被誤判的合法家具 = 0** 才接線,風險由 Medium 實測降為 Low。

### 🏛️ Column Detection(Phase 3-3)
**`COLUMN_TOLERANCE_MM = 300`,單位是「穿入深度 mm」**(與 `WALL_TOLERANCE_MM`
的 mm² 不同)。

容差不是猜的。34 案 941 件家具實測:

| 項目 | 數值 |
|---|---|
| 貼牆合法壓柱的家具 | **283 件(30.1%)** |
| 最深穿入 | **175 mm** |
| 理論上限(柱半 250 − 內牆半厚 60) | 190 mm |
| 形心落在柱內(真錯誤) | **0** |

本 repo 的柱 **100% 藏在牆內**,家具貼牆必然壓到柱伸進室內的半邊。若用
`area > 100` 或把容差設 150mm,那 283 件合法家具會全被誤判。取 300mm
(柱 500 × 0.6)高於合法上限仍留 110mm 裕度。

另提供 `column_contacts()` 報表,列出**所有**家具×柱接觸(含合法的),
標示穿入深度、面積與是否超標。

### 🔧 Column Resolver(Phase 4)
`try_move()` 新增**避柱守衛**——柱不能放進面積判定的 `blockers`(那會封殺
30% 合法貼牆落點),而是與牆對稱地走獨立的穿入深度判定。三道守衛
(牆突出 / 柱穿入 / blockers 重疊)在**同一個候選位置**同時檢查,
所以修好柱碰撞不可能製造新的牆、天井或樓梯碰撞。

柱碰撞**只移動、不丟棄**:修不動就保留家具原位並標記
`ResolveReport.unresolved_column`,交給人判斷——結構柱衝突不該讓家具憑空消失。

## 品質驗證

### Regression = 0
| 指標 | 結果 |
|---|---|
| Benchmark(四階段前後) | **34/34 · 20 pass / 14 warn / 0 fail — 逐字相同** |
| 生成後再跑一次 `resolve()` | **100/100 層完全 no-op** |
| 殘留碰撞(牆 · 天井 · 樓梯 · 柱) | **0 · 0 · 0 · 0** |
| `unresolved_column` | **0** |
| 測試 | **501 passed**(v0.5 為 472) |

最後一列的「二次 resolve 全 no-op」是最強的 regression 證據:四類障礙全開之後,
對每一張生成完的圖再跑一次修復,100 層全部完全不動。

### Benchmark 通過
34 案巡檢(12 單層 / 18 透天 / 4 天井)全數生成成功,狀態分佈與 v0.5 完全一致。

### DXF / PNG 驗證完成
- **DXF 100/100** — 存檔 + ezdxf 重讀 + 實體/圖層檢核
- **PNG 100/100** — matplotlib 算圖
- `report.html` 自包含報告產出正常

### 實際修掉的缺陷
**B03(18×13m 透天兩層)** 1F 餐桌原本穿過一道實心牆、伸進儲藏室約 600mm
(`validate_spec` 一直沒抓到),現在生成當下自動東移 300mm 修回餐廚。

## 相容性

`layout_generator` 只接線兩處 `resolve_collisions()`
(`_validate_or_raise` 與 `generate_floor_plan`),**生成邏輯零改動**;
Room Program、Generator 皆未修改。既有 API 與輸出對合格案例逐字不變。

## 已知限制

- **Column Resolver 目前不會被觸發**:真實圖面的柱全藏在牆內,伸進室內最多
  250mm < 容差 300mm,故超標壓柱在現有案例中不可能發生。它是為未來
  **獨立柱**(開放空間落柱、中島腳)預留的防線。
- **Elevator / Shaft 未納入**:Shaft 在本 repo 是「管道牆」= 牆,已被 Phase 2
  覆蓋;Elevator 有 `spec.elevators` 資料但無 provider,且不在 benchmark 範圍。
- `engine` 的「換對方讓開」後備路徑因 `blockers2` 未排除 WALL,自 Phase 2 起
  實質為死碼(不影響現行行為,待後續清理)。
- `geometry.py` 有一行重複的 `VOID` import(dead code)。
- `validate_spec` 與 `collision_problems` 的家具檢核邏輯仍重複(刻意的過渡設計,
  待收斂為單一來源)。
