# v0.7.0 — Layout Analysis Stack

**發布日期:2026-07-22**

v0.6 讓引擎能把家具**擺對**;v0.7 讓引擎能回答一個更難的問題:
**這張圖到底好不好?**

新增七層分析堆疊——驗證、連通、動線、評分、常規、微調、比較——並訂下一條
貫穿全部的規矩:**分析層預設唯讀,不接進生成流程**。因此整個 v0.7 期間,
Benchmark 維持 `34/34 · 20 pass / 14 warn / 0 fail` **逐字不變**。

---

## 主要功能

### ① Layout Validation
多邊形是否封閉、房間是否重疊、是否有孤立房、門是否落在房間邊界、走道是否中斷。

### ② Connectivity Graph
同一份 spec 的四種視角:

| 圖 | 意義 |
|---|---|
| Adjacency | 房間實體相鄰(共用邊界夠長) |
| Room | 房間可通行,邊有 `door` / `open` 兩種型別 |
| Space | 用 `open` 邊把房間併成開放空間,Space 之間以門相連 |
| Door | 每扇門服務哪些房間(內門 / 對外門 / 孤兒門) |

實測 100 層:door 邊 495 · open 邊 142 · 對外門 34 · **孤兒門 0**。

### ③ Corridor Analyzer
走道寬/長、Bottleneck、Dead End、Walking Distance(Dijkstra,經實際通行點轉折)、
Longest Path。實測最遠步行 median 14.5m · max 30.7m。

### ④ Layout Scoring
七面向加權評分,權重可調:

```
total = Σ(subscore × weight) / Σ(weight)
connectivity 2.0 · circulation 1.5 · privacy 1.0 · lighting 1.5
utilization 1.0 · furniture 1.0 · collision 2.0
```

### ⑤ Constraint Engine
五條可登錄的設計常規;條件不成立自動列入 `skipped`,不會被誤判成通過或違反。

| 規則 | 嚴重度 | 100 層觸發 |
|---|---|---|
| 臥室不可直接面廚房 | error | 0 |
| 衛浴不可直接面餐廳 | error | **6** |
| 入口不可直視馬桶 | error | 0 |
| 臥室避免緊鄰廚房/餐廳 | warn | 12 |
| 廚房應接近餐廳 | warn | 5 |

### ⑥ Layout Optimizer(唯一會改格局的分析層)
`propose → verify → accept / revert`,候選一律在 deepcopy 上試算,通過安全閘門
(validate 全過 · 連通過 · **碰撞不增加** · 房間不被壓扁)且確實更好,才把**那一個**
改動寫回。**刻意不接進生成流程**。

### ⑦ Optimization Benchmark
Before → Optimize → After,輸出 JSON + 前後 DXF/PNG。8 層實測:
**改善 5 · 退步 0 · 平均 +0.015**。

## 這一版最重要的心得:量測救回了四個模組

v0.7 有四次「第一版看起來能跑,實際上是廢的」,全靠先量測才發現:

| 模組 | 第一版的問題 | 診斷 | 修正後 |
|---|---|---|---|
| Connectivity | 34 案 **253 次孤立房誤報** | 開放式餐廚是「有洞口、無 DoorPlacement」,只看 `spec.doors` 看不到 | 0 誤報 |
| Privacy 評分 | 中位數 **0 分** | 把「家庭廳」當公共空間,懲罰透天臥室層的正常設計(75 次) | 中位數 60,有鑑別度 |
| Constraint | 兩條 warn 觸發率 **88% / 77%** | 臥室與客廳共牆在小宅無可避免;「接近」不等於「必須開門相通」 | 21% / 38% |
| Optimizer | 20 層 **0 改動** | 候選漏掉 L 形房;最窄房邊用絕對門檻造成全部連坐否決 | 9/20 層找得到安全改善 |

共同教訓:**憑感覺設的門檻幾乎都是錯的**,而**一條在 88% 合格圖上都會叫的規則
等於沒有規則**。這兩條已寫進 `docs/DEVELOPMENT_GUIDE.md`。

## 相容性

分析層全部是新增模組,**未修改 Generator、Layout Generator、Room Program**。
唯一對既有程式的改動是替四個既有 Report 補上 `to_dict()` / `to_json()`
(純新增方法),以及 `layout_validation` 改為委派 `connectivity`(行為保持不變,
原 13 個測試全數照過)。

## 已知限制

- **Optimizer 增益極小**(+0.01 分),且三種微調**修不掉 constraint error**:
  實測把門沿牆掃過所有位置(S06 690 個、S08 839 個)都無法改變它開向哪一間,
  那需要改拓樸,不屬於「微調」。生成器輸出已接近這三種操作的局部最佳。
- **`room_rotation` 一定被否決**:軸對齊鋪滿平面上旋轉單一房間必然破壞鋪滿。
  保留是為了日後自由平面,已寫成測試釘住,避免誤以為它有在作用。
- **L 形走道寬度被高估**(最小外接矩形近似),只會漏報不會誤報。
- **6 個樓層有「衛浴直開餐廳」**:分析層只回報,修正屬 Generator 範疇。
- `validate_spec` 與 `collision_problems` 的家具檢核仍重複(刻意的過渡設計)。
- v0.6 遺留:`engine` 的「換對方讓開」後備路徑因 `blockers2` 未排除 WALL,
  實質為死碼(不影響現行行為)。
- 天井版 Living Overflow 尚未套用(P01~P04)。
